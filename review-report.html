06 / Findings

06具体发现 · 按 Blocker → Major → Minor 排序

CR-002BLOCKERAuthoritativeBoundary · RuntimeCorrectness
server/claude-bridge.js · 第 71-110 行 callClaude / 第 114-128 行 checkClaudeAvailable
证据：spawn('claude', args, { shell: process.platform === 'win32' }) — Windows 上开 shell + 用户可控 model 字段从 HTTP POST body 解析。即便 args 是数组，shell:true 让 Node join 后交给 cmd.exe 解析，model: "x'; calc.exe; '" 类输入可执行任意命令
修复：参照 server/feishu-bot-daemon.js 的 spawnLark 模式 — Windows 用 spawn('cmd.exe', ['/c', 'claude.cmd', ...args], { shell: false, windowsHide: true })。args 不经 shell 解析，安全。同步修 checkClaudeAvailable
CR-001MAJORDuplication · SharedStructure
src/stores/useCanvasStore.js · 行 55, 120, 186, 219, 242, 298, 318, 384, 404, 441, 573, 1206, 1422, 1518, 1630, 1755, 1811, 1861（共 18 处）
证据：${prefix}-${Date.now()}-${Math.random().toString(36).slice(2, 8)} 模式直接重复 18 次。Math.random base36 截断 6 字符 = 36⁶ ≈ 2.2 亿组合，加 ms 时间戳协作场景下碰撞概率极低，但仍为概率事件
修复：抽 src/utils/genId.js：export const genId = (prefix) => `${prefix}-${crypto.randomUUID()}`。替换全部 18 处。crypto.randomUUID 浏览器原生，2.7×10³⁸ 组合，协作 ID 永不碰撞
CR-003MAJORConcurrency · RuntimeCorrectness
src/collab/yjsSync.js · 第 60-71 行 attachYjsSync 启动握手
证据：
provider.once('sync', () => { synced = true; onSynced() })
setTimeout(() => { if (!synced) onSynced() }, 1500)
1.5s 兜底定时器和 'sync' 事件几乎同时触发时，!synced 检查与 synced=true 之间无原子保护 — onSynced 可能双执行。yjs map.set 幂等保正确性，但 _suppressLocalPush=false 二次回写不雅，stashedNodes 也会 push 两次
修复：增加 let onSyncedCalled = false，onSynced 入口先 return-if-called。或更优雅 const finalize = once(() => finalizeStartup(stashedNodes, stashedEdges))
CR-004MAJORPerformance
src/collab/yjsSync.js · 第 76-86 行 store.subscribe + 第 122-148 行 pushLocalToYjs
证据：每次 store.subscribe 触发都跑全量 diff — 对 N 节点做 2N 次 JSON.stringify（prev vs plain）。React Flow 拖动节点 60 fps，N=100 时每秒 12000 次 JSON.stringify。注释解释"立即推送避免节流期间被远端覆盖"，但这是伪两难（增量 dirty 跟踪可同时满足两点）
修复：(a) reference 比较代替 JSON.stringify：store.subscribe 的 prev/next nodes 数组同引用就跳过；(b) trailing-edge throttle 50ms 保证最终态推送；(c) 维护 dirtyNodeIds: Set 只 diff 变化节点
CR-005MAJORAuthoritativeBoundary
src/collab/yjsClient.js · 第 50 行 getProvider()
证据：export function getProvider() { return _provider } — 把内部 WebsocketProvider 实例直接暴露成公共 API。yjsClient 是 Yjs 连接的权威边界，其上层调用方应仅通过 startSync / stopSync / setLocal* / onAwarenessChange 操作。getProvider 让任何调用方能拿到 provider 后绕过受控入口
修复：改成私有 _getProvider，加 @internal 注释；如有外部需求，应在 yjsClient 暴露 narrow API（onSync(handler) / onConnectionError(handler)）
CR-009MAJORSharedStructure · API
src/stores/useCanvasStore.js · 第 651-672 行 changeNodeType
证据：preserved 字段硬编码 7 个公共字段，仅 noteNode 有 description→content 映射。但实际节点类型 11 种，各自核心字段不同：
· taskNode → conceptNode：丢 hermesAssignee / progress / agentMode
· bookmarkNode → conceptNode：丢 url
· imageNode → conceptNode：丢 src / alt
修复：建立 NODE_TYPE_FIELD_MAP，每个类型声明 input / output 字段。changeNodeType 用 mapping 矩阵转换，未映射字段保留到 node.data._archived 以备回切。当前实现对 demo 够用，但协作场景"三人同步把 task 改成 concept"会让其他两人看到"任务进度无故消失"
CR-006MAJORRuntimeCorrectness · Cleanup
server/y-ws-server.js · 第 51-61 行 setPersistence bindState
证据：bindState 内 require('yjs') 重复 3 次，且第 51 行重新 require('y-websocket/bin/utils') 取 setPersistence — 第 36 行已经 require 过同一模块取 setupWSConnection。两次 require 在 cache 命中下无副作用，但表达不一致
修复：顶部统一 const Y = require('yjs')、const { setupWSConnection, setPersistence } = require('y-websocket/bin/utils')
CR-007MAJORAPI · RuntimeCorrectness
src/services/aiProvider.js · 第 35-38 行 mock 分支
证据：case 'mock': default: return '' — 业务调用方期望 string 后跑 JSON.parse 或正则提取；空字符串让 parse 失败或拿到空数组，调用方误以为"AI 没找到概念"，实际是没调 LLM
修复：(a) throw new Error('mock provider 未实现 LLM 调用') 让调用方走兜底规则解析；或 (b) 返回明确占位 JSON.stringify({ _mode: 'mock', concepts: [], relations: [] })
CR-010MINORAuthoritativeBoundary
src/collab/useCollabSession.js · 第 14-15 行 import + 第 23-31 行 useEffect
证据：useCollabSession 同时 import {startSync, stopSync, setLocalUser} 自 yjsClient 和 {attachYjsSync, detachYjsSync} 自 yjsSync。这两个文件应有统一 facade — useCollabSession 是上层，本不应同时依赖两个低层
修复：在 src/collab/index.js 加 facade：export const startCollab = (roomId, user) => { startSync(roomId, {user}); setLocalUser(user); attachYjsSync() } 与 stopCollab
CR-008MINORDuplication
src/services/aiProvider.js · 第 83-97 行 callVpsProxy + 第 103-117 行 callClaudeCli
证据：两个函数共享 fetch + !resp.ok 抛错 + data.ok 检查 + 返回 data.text 模式。callOpenAiLike 因协议不同合理独立
修复：抽 postChatJson({url, body, errPrefix}): Promise<string> 内部 helper

08值得保持的优秀设计

Collab 子系统结构

6 文件各拥单一关注点（连接/桥/Awareness/凭据/生命周期/UI），是本次评审最佳实践示范

启动序列设计

stash + suppress + sync 等待 + 决策推/拉 + 解 suppress 五步法清晰，注释足以让接棒 cc 立刻上手

ORIGIN_LOCAL transaction

用 Y.Transaction.origin 区分本地推/远端推，避免双向 sync 回弹环；单一所有者，单一标识

AI provider 注册表

PROVIDER_PRESETS 数组 + getActiveProvider 派生 — 注册表模式纯净，新增 provider 仅改一处

Awareness 节流策略

PresenceLayer 鼠标 30fps 节流 + trailing-edge 兜底，远端光标不抖；视口变化 200ms forceUpdate 让坐标重算

y-ws-server token 鉴权

环境变量驱动可选 token，?token=<v> 与 X-Header 双通道；upgrade 阶段拒绝，无 doc 泄漏
